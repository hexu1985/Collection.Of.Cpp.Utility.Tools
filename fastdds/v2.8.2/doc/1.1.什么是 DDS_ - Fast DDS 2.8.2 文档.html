1.1. 什么是DDS?¶

数据分发服务(DDS) 一种以数据为中心的通信协议,用于分布式软件 应用通信。 它描述了通信应用程序编程接口(API)和通信语义,以实现 数据提供商与数据消费者之间的通信。

由于它是以数据为中心的出版订阅(DCPS)模型,因此在其中定义了三个关键应用程序实体。 实现:发布实体,定义信息生成对象及其属性; 定义信息消耗对象及其属性的订阅实体;和配置实体 定义作为主题传输的信息类型,并创建发布者和订阅者 其服务质量(QoS)属性,确保上述实体的正确性能。

DDS 使用 QoS 来定义 DDS 实体的行为特征。QoS由个别QoS策略组成 (源自 QoSpolicy 的类型对象)。这些在政策中描述。Policy

1.1.1. DCPS概念模型¶

在DCPS模型中,定义了四个基本元素,用于开发通信应用程序系统。

  • 出版商。 它是DCPS实体,负责创建和配置其实现的DataWriters。 DataWriter 是负责实际发布消息的实体。 每个都将有一个分配的主题,根据该主题发布消息。 有关详细信息,请参阅发布商。

  • Subscriber订阅者。 它是DCPS实体,负责接收根据其订阅的主题发布的数据。 它为一个或多个 DataReader 对象提供服务,这些对象负责传达新数据的可用性。 到应用。 有关详细信息,请参阅订阅者。

  • 主题。 是绑定出版物和订阅的实体。 它在DDS域中是独一无二的。 通过 TopicDescription,它允许出版物和订阅的数据类型的统一。 有关更多详细信息,请参阅主题。

  • 域名。 这是用于链接所有出版商和订阅者的概念,属于一个或多个应用程序, 在不同主题下交换数据。 参与域的这些单独应用程序称为 DomainParticipantDomainParticipant。 DDS 域由域 ID 标识。 DomainParticipant 定义域 ID 以指定它所属的 DDS 域。 具有不同 ID 的两个 Domain 参与者不知道彼此在网络中的存在。 因此,可以创建多个通信渠道。 这适用于涉及多个 DDS 应用程序的场景,以及各自的 DomainParticipant 。 相互通信,但这些应用程序不得干扰。 DomainParticipant 充当其他 DCPS 实体的容器,充当工厂 PublisherSubscriber发布者、订阅者和主题实体,并在域中提供管理服务。 有关详细信息,请参阅 Domain。

1.2. 什么是RTPS?¶

实时发布订阅(RTPS)协议,开发到 支持DDS应用,是一种出版-订阅通信中间件 超过最佳努力的运输,如UDP / IP。此外,Fast DDS 还支持 TCP 和 共享内存(SHM)传输。

它旨在支持单播和多播通信。

在从 DDS 继承的 RTPS 的顶部,可以找到域,它定义了一个单独的通信平面。 多个领域可以同时独立共存。 域包含任意数量的RTPSParticipants,即能够发送和接收数据的元素。 为此,RTPSParticipants 使用他们的 EndpointsEndpoints :

  • RTPSWriter : 端点能够发送数据。

  • RTPSReader : 端点能够接收数据。

RTPSParticipant 可以具有任意数量的写入器和读卡器端点。

RTPS 高级架构¶

沟通围绕主题展开,主题定义和标记正在交换的数据。 主题不属于特定参与者。 参与者,通过RTPSWriters,对在主题下发布的数据进行更改,并通过RTPSReaders进行更改 接收与其订阅的主题相关的数据。 通信单元称为 Change,它表示在 Topic 下写入的数据中的更新。 RTPSReaders/RTPSWritersRTPSReaders/RTPSWriters在其历史记录上注册这些更改,History该数据结构可作为缓存 最近的变化。

在 eProsima Fast DDS 的默认配置中,当您发布变化通过 RTPSWriter 端点, 以下步骤发生在幕后:

  1. 因变化添加到 RTPSWriter 的历史缓存中。

  2. RTPSWriter将更改发送给它所知道的任何RTPSReaders。

  3. 接收数据后,RTPSReaders 会更新其历史记录缓存。

但是,Fast DDS 支持多种配置,允许您更改 RTPSWriters/RTPSReaders 的行为。 RTPS实体默认配置的修改意味着数据交换流之间的变化 RTPSWriters 和 RTPSReaders。 此外,通过选择服务质量(QoS)策略,您可以影响这些历史缓存在几个中的管理方式。 方式,但通信循环保持不变。 您可以继续阅读 RTPS 层,了解有关快速 RTPS 协议实现的更多信息 DDS。
