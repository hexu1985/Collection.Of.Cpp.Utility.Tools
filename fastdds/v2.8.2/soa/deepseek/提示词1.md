我目前做如下设计：
首先是服务名维护在fastdds的TOPIC里，不同的服务对应不同的TOPIC，以服务ServiceA为例，在fastdds中定义soa.rpc.ServiceA.request和soa.rpc.ServiceA.response这两个TOPIC，请求和响应的TOPIC对应的idl定义如下：
module fastdds_soa {

enum ErrorCode {
    SUCCESS,
    SERVICE_NOT_AVAILABLE,
    METHOD_NOT_REGISTER,
    REQUEST_TIMEOUT
};


struct RPC_Header {
    string method_name;
    string client_id;
    long session_id;
    long request_id;
};

struct RPC_Request {
    RPC_Header header;
    string request_payload;
};

struct RPC_Response {
    RPC_Header header;
    ErrorCode error_code;
    string response_payload;
};

};  // fastdds_soa
每个请求header里包含method_name，区分同一个服务下的不同rpc调用，header里包含client_id来区分不同的rpc client，session_id和request_id分别对应client的生命周期和rpc调用的生命周期。
