`b2` 命令行参数类型主要分为三种：**选项（Option）**、**属性（Property）** 和**目标（Target）**，这三者可以任意顺序使用。`b2` 的具体命令在源码编译成功后是在源码根目录下的一个可执行程序，常在当前目录调用（如键入 `./b2`）。

下面按这三类并补充一组常用控制命令为您详细解释：

*   **⭐ 核心概念：选项 (Options)**
    用于控制 `b2` 构建系统的行为，以单个或两个连字符开头 `-` 或 `--`。
    | 选项示例 | 作用 |
    | :--- | :--- |
    | `-a` | **全部重建**。忽略时间戳，强制重新编译所有目标。 |
    | `-j [N]` | **并行构建**。同时运行 N 个命令，充分利用多核CPU加速编译，如 `-j4`。 |
    | `-n` | **模拟运行**。只打印将执行的命令，而不实际运行，用于调试和预览。 |
    | `-q` | **遇错即停**。在首个错误处停止构建，适合快速修复问题。 |
    | `--clean` | **清理目标**。清除当前和子项目的所有构建目标。 |
    | `--debug-building` | **调试：显示构建细节**。打印正在构建的目标及其使用的编译属性。 |
    | `--help` | **查看帮助**。显示通用的命令行帮助信息。 |

*   **⭐ 核心概念：属性 (Properties)**
    指定“要构建什么”，通过 `特性=值` 的格式定义，用以控制库的生成变体。
    | 属性示例 | 作用 |
    | :--- | :--- |
    | `variant=debug/release` | **构建版本**。指定生成调试版还是发布版。也可简写为 `debug` 或 `release`。 |
    | `link=static/shared` | **链接方式**。指定生成静态库 (`.a`) 还是动态库 (`.so`)。 |
    | `runtime-link=static/shared` | **运行时链接**。指定C/C++运行时库的链接方式。 |
    | `threading=single/multi` | **线程支持**。指定库是否支持多线程。 |
    | `address-model=32/64` | **地址模型**。指定生成32位或64位的库。 |
    | `toolset=gcc/msvc` | **指定工具集**。明确使用哪个编译器。 |

*   **目标 (Targets)**
    所有既不是 `选项` 也不是 `属性` 的参数都被视为 **`目标`**，用于决定构建什么，可以是一个库、可执行文件或一个特定行为。
    *   `stage`：只构建库文件（`.a`/`.so`），并输出到指定目录（默认为 `./stage`）。
    *   `install`：**安装**。构建并将头文件、库文件等安装到系统目录（通常需要 `sudo`）。
    *   `--with-<library>` / `--without-<library>`：白名单/黑名单，只编译/排除特定库。

*   **常用控制参数详解**
    以下参数常用于控制库的生成方式、存储路径等细节，"常用控制参数详解"部分更像是对前文"目标和选项"的补充和细化。
    | 参数示例 | 作用 |
    | :--- | :--- |
    | `stage` / `install` | **构建动作**。前者只生成库文件 (`stage`)，后者还会安装头文件。 |
    | `--stagedir=<DIR>` | **输出目录**。配合 `stage` 使用，更改库文件的存放路径，默认为 `./stage`。 |
    | `--prefix=<PREFIX>` | **安装路径**。配合 `install` 使用，指定安装的根目录，Unix下默认为 `/usr/local`。 |
    | `--build-type=minimal/complete` | **构建集**。`minimal` (默认) 构建最小集；`complete` 则构建所有可能的变体组合。 |
    | `--layout=versioned/tagged/system` | **命名布局**。控制生成库文件的命名规则及头文件安装路径，直接影响多版本共存能力。 |
    | `--buildid=ID` | **构建标识**。为生成的库文件名添加自定义后缀，便于区分不同配置或版本的构建产物。 |
    | `--build-dir=DIR` | **构建目录**。指定构建过程中产生的中间文件存放位置，默认在源码目录内。 |
    | `--show-libraries` | **显示库列表**。列出所有需要编译安装的Boost库，然后退出。 |

### 📊 命名布局（`--layout`）对比

`--layout` 是比较特殊且重要的一个**选项**，它决定了生成库文件的命名规则和安装路径结构，直接影响多版本、多编译器共存的便利性，具体规则如下：

| 布局策略 | **库文件命名特点** | **头文件安装路径** | **适用场景** |
| :--- | :--- | :--- | :--- |
| **`versioned`** | **名称包含版本号、编译器及构建属性** | `include/boost-<version>/` | Windows系统下的默认值，适合在同一系统上保留多个Boost版本。 |
| **`tagged`** | **名称包含构建属性（如`-mt-mt`），但不含版本和编译器信息** | `include/boost/` | 使用**同一个编译器**构建多个变体（如debug/release）时使用。 |
| **`system`** | **名称不含版本号或编译器标识** | `include/boost/` | Unix/Linux系统的默认值，系统包管理器集成时使用。 |

你可以将这些参数组合起来，实现精细化的构建控制。根据前一问中编译Boost 1.65.1的命令 `./b2 -j4 --build-type=complete --layout=tagged toolset=gcc stage`：

*   **`-j4`**：是一个**选项（Options）**，用来定义构建行为，加快了编译速度。
*   **`--build-type=complete`** 和 **`--layout=tagged`**：都是用于特殊行为的**选项（Options）**，前者指定了构建完整的库变体集合，后者则控制了库文件的命名规则，以便区分不同的构建配置。
*   **`toolset=gcc`**：是一个**属性（Properties）**，指明了要使用的编译器。
*   **`stage`**：是一个**目标（Targets）**，指明了要执行的动作，这里表示“只生成库文件”。


### b2编译命令和b2安装命令是分开的吧？

是的，`b2`的**编译**和**安装**命令通常是分开的，或者说通过不同的**目标（target）**来区分。

- **编译命令**：`./b2 stage`  
  只生成库文件（`.a` / `.so`），输出到`stage/lib`目录，不会拷贝头文件或安装到系统路径。

- **安装命令**：`./b2 install`  
  会先编译库文件，然后将**头文件**和**库文件**一起复制到指定前缀目录下（默认为`/usr/local`，可通过`--prefix`修改），通常会需要`sudo`权限。

你也可以在同一命令中只做编译（stage），后续手动复制，也可以直接使用`install`一条命令完成编译+安装。所以从命令角度看，它们是分开的，通过不同的目标名控制。



### 指定安装目录，是在b2 stage里指定，还是在b2 install里指定？还是两个都要指定。

分开的。具体来说：

- **`b2 stage`**：只生成库文件（不安装），输出目录用 **`--stagedir=`** 指定。  
  例如：`./b2 stage --stagedir=/my/custom/lib`

- **`b2 install`**：编译并安装（头文件+库文件），安装根目录用 **`--prefix=`** 指定。  
  例如：`sudo ./b2 install --prefix=/my/install/prefix`

**所以答案是**：在 `b2 stage` 里指定的是**临时输出目录**；在 `b2 install` 里指定的是**最终的安装路径**。两者不会也不应该使用同一个参数。如果你只做 `stage`，就用 `--stagedir`；如果你做 `install`，就用 `--prefix`。不必两个都指定。
